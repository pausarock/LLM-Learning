## 2. 核心概念讲解

### 2.0 从「能跑」到「可用」：第 6 天的意义

day5 让你拿到了第一版结果；day6 解决三个真实问题：

1. **为什么贵**：对 token 不敏感，prompt 越积越长；  
2. **为什么慢**：上下文太长、模型太大、流式未观测首字时间；  
3. **为什么不稳**：没有会话管理、缺少指标，问题复现困难。

今天你要把脚本升级成「可持续迭代」的小工具。

---

### 2.1 Token、上下文窗口与成本

核心关系（足够工程化）：

- 输入 token 越多，**prefill** 成本越高，延迟越大；  
- 输出 token 越多，**decode** 时间越长，费用越高；  
- 多轮对话如果把全部历史都带上，会线性甚至更快地变贵变慢。

这与 `llm-terminology.md` 的 **KV Cache / Prefill-Decode** 概念是同一件事在 API 层的表现。

**工程底线**：每次请求都要有「预算意识」：

- 预算一：最大输入（上下文）  
- 预算二：最大输出（`max_tokens`）  
- 预算三：总请求成本（可估算）

---

### 2.2 上下文管理：你必须主动裁剪

最小策略（推荐今天实现）：

1. 始终保留 1 条 `system`；  
2. 用户/助手对话只保留最近 N 轮（例如最近 4～8 轮）；  
3. 如果有 token 估算器，再按预算裁剪到阈值内。

常见误区：

- 把完整历史无限堆入 `messages`；  
- 每轮都附长篇「固定说明」导致输入膨胀；  
- 不区分「必须记住」与「可丢弃」信息。

如果场景需要长期记忆，应转向「摘要记忆 + 外部检索（RAG）」而不是无脑塞上下文。

---

### 2.3 可观测性：至少记录 4 个指标

你不需要一上来接入复杂监控，先记住最小四件套：

1. **TTFT**（time to first token，首包时间，流式体验核心）  
2. **总耗时**（端到端）  
3. **输出长度**（字符或 token）  
4. **重试次数**（稳定性）

可选再加：

- 请求 id（若服务端提供）  
- 模型名  
- 流式/非流式标记  
- 错误码分布

有这些数据，你才能判断问题在「网络、服务、模型大小、prompt 长度」哪一层。

---

### 2.4 参数与稳定性：不是越多越好

在 day5 参数基础上，day6 建议建立「默认配置」：

- 问答类：`temperature=0.2~0.4`  
- `top_p=1`（先不和 temperature 同时大幅联调）  
- `max_tokens` 由产品需求倒推（简答 128，讲解 512 等）

稳定性实践：

- 先固定模型和参数，先控变量；  
- 记录失败请求输入长度与参数，避免「复现不了」；  
- 对高风险任务增加输出约束（格式、长度、步骤）。

---

### 2.5 最小 CLI 聊天器设计（今天交付物）

建议文件：`day6_cli_chat.py`。

功能清单：

1. `python day6_cli_chat.py` 启动后进入循环输入；  
2. 输入普通文本 -> 发起流式请求 -> 实时打印回答；  
3. 支持命令：  
   - `/clear`：清空历史（保留 system）  
   - `/exit`：退出程序  
4. 每轮结束打印简短统计：  
   - `ttft_ms` / `elapsed_ms` / `chars` / `retries`

这就是你在面试/演示里可以现场跑的工程样例。

## 5. 练习题

### 必做 1：实现会话管理容器

目标：维护 `messages`，结构固定：

- 首条 `system`  
- 后续成对追加 `user` / `assistant`

要求：

1. 提供 `append_user(text)` 与 `append_assistant(text)`；  
2. 提供 `clear_history()`（仅保留 system）；  
3. 提供 `trim_history(max_turns)` 保留最近 N 轮。

<details>
<summary>参考答案要点</summary>

- 可用 `list[dict]` 存消息  
- `max_turns` 以「user+assistant」为 1 轮  
- 裁剪前记得保留 system 在索引 0

</details>

---

### 必做 2：流式统计 TTFT 与总耗时

目标：每轮回答后输出：

- `ttft_ms`  
- `elapsed_ms`

要求：

1. 请求开始时记录 `t0`;  
2. 首个非空 token 到达时记录 `t_first`;  
3. 完成后计算总耗时。

<details>
<summary>参考答案要点</summary>

- `ttft_ms = (t_first - t0) * 1000`  
- `elapsed_ms = (t_end - t0) * 1000`  
- 如果无 token（错误中断），TTFT 记为 `None` 并打错误日志

</details>

---

### 必做 3：成本意识实验（轻量）

选同一个问题，做两组对比：

1. 短上下文（仅 system + 当前 user）  
2. 长上下文（附加多轮历史）

记录并比较：

- 总耗时  
- 输出质量感受  
- （若响应有 usage 字段）输入/输出 token

写 3 句话总结：何时「多历史」值得，何时是浪费。

---

### 必做 4：CLI 命令与异常处理

要求：

1. `/exit` 安全退出；  
2. `/clear` 后下一轮仍能正常回答；  
3. 网络错误时不崩溃，提示后继续下一轮输入；  
4. 记录本轮是否发生重试。

---

### 选做 5：按预算裁剪（进阶）

如果你接入 token 估算器（如 `tiktoken`），尝试：

- 设 `max_input_tokens`，超出就从最旧轮次开始删除；  
- 比较「按轮裁剪」与「按 token 裁剪」的差异。

---

### 选做 6：结构化日志文件

把每轮统计写入 `logs/day6_chat.jsonl`，每行一个 JSON。  
字段建议：`ts`, `model`, `ttft_ms`, `elapsed_ms`, `retries`, `input_chars`, `output_chars`, `ok`。
